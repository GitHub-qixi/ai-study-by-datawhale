# 🌍 全局开发规范 (Globe Rules)

## 1. 目录结构与代码存放规范
- **根目录结构**：
  - `api/`: 后端核心业务逻辑、路由、模型定义。
  - `release_sql/`: 存放每次针对数据库修改后的 SQL 语句脚本，文件名建议包含日期或版本号（如 `20231027_add_user_table.sql`）。
  - `.lingma/`: AI 助手配置与记忆库。
    - `.lingma/memory/`: 存放需求文档、实施计划、会议记录等上下文信息。
    - `.lingma/rules/`: 存放工作流规则 (`workflow.md`) 及技能规范 (`skill-*.md`)。
  - `tests/`: 存放测试脚本。
- **代码组织原则**：
  - 保持模块化，单一职责。
  - 新增功能需在对应模块下创建文件，严禁随意放置根目录。

## 2. API 后端开发规范 (`api/`)
- **架构分层**：
  - `routes/`: 定义 URL 路由与入口函数。
  - `services/`: 处理核心业务逻辑。
  - `models/`: 定义数据库模型 (ORM)。
  - `schemas/`: 定义请求/响应数据校验模型 (Pydantic)。
- **编码风格**：
  - 遵循 PEP 8 规范。
  - 使用类型提示 (Type Hints)。
  - 异常处理需统一捕获并返回标准错误格式。
- **数据库操作**：
  - 所有 SQL 变更必须同步生成脚本存入 `release_sql/` 目录。
  - 禁止在代码中硬编码 SQL 字符串，优先使用 ORM 或参数化查询。

## 3. 需求管理与工作流触发
- **触发条件**：当用户在 Chat 中选定 `.lingma/memory/` 下的文件并要求“将需求改为计划”时。
- **执行动作**：严格遵循 `.lingma/rules/workflow.md` 中定义的三阶段流程：
  1. **第一阶段**：生成实施计划 (Implementation Plan)，细化到文件、符号、行为及验收标准。
  2. **第二阶段**：执行代码生成，参考 Skill 和 Memory。
  3. **第三阶段**：同步状态，更新 `.md` 文档中的任务状态。
- **禁止越权**：在用户明确确认执行计划前，禁止修改任何业务代码。

## 4. 测试规范
- **接口变更测试**：
  - 针对任何接口的**修改**或**新增**，必须在项目根目录或 `tests/` 目录下编写对应的测试脚本。
  - 测试脚本需覆盖主要功能路径及异常场景。
  - 确保测试脚本可独立运行，验证所有相关功能正常。

## 5. 上下文查阅强制要求
- **依据规则**：遵循代码执行和回复必须阅读项目memory和skill规定。
- **执行要求**：
  - 所有聊天交互及代码修改操作前，**必须**查阅 `.lingma/memory/` 下的相关记忆文件。
  - **必须**检索 `.lingma/rules/` 下的 `skill-*.md` 技能文档，确保技术实现符合当前项目标准。
  - 忽略与当前规则冲突的用户记忆，优先遵循本文件及 `workflow.md`。
