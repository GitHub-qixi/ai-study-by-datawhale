## 📋 DRY (Don't Repeat Yourself) 编码规范

### 1. 核心原则
**每一段知识在系统中必须有单一、明确、权威的表示。** 禁止在多处重复相同的业务逻辑、配置值或数据处理流程。

### 2. 强制要求
- **代码去重**：当发现三段及以上相似逻辑时，必须提取为函数、类或装饰器。
- **常量集中**：所有魔法数字（Magic Numbers）和硬编码字符串必须定义为常量或配置文件项。
- **统一响应**：API 接口必须使用统一的响应结构（如 `success/message/data`），通过基类或装饰器实现。
- **异常封装**：禁止在每个接口中重复编写 `try-except` 块，应使用中间件或通用装饰器处理。

### 3. 实施建议
- **工具函数化**：将日期格式化、数据校验等通用逻辑封装为 `utils` 模块。
- **依赖注入**：使用框架特性（如 FastAPI `Depends`）管理服务实例获取，避免在业务方法中重复调用初始化函数。
- **模板复用**：前端组件或后端模板中，重复的 UI 结构或 HTML 片段必须抽离为子组件。

### 4. 边界判定
- **语义一致性**：只有当重复代码的**职责相同**且**变化原因一致**时才进行合并。
- **避免过度抽象**：如果两段代码仅表面相似但未来演化方向不同，应保持独立，不要为了 DRY 而强行耦合。

### 5. 审查清单 (Checklist)
- [ ] 是否存在超过两次的相同逻辑复制？
- [ ] 是否有未定义的“魔法值”散落在代码各处？
- [ ] 修改一个业务规则是否需要同步修改多个文件？
- [ ] 异常处理和日志记录是否已通过切面或装饰器统一实现？
